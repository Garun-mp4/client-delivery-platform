# UX foundation Garun Workspace

Статус: реализовано в Milestone 06.5
Дата: 2026-07-18

## 1. Пользовательское намерение

Владелец открывает Garun Workspace между рабочими задачами и должен за несколько секунд понять:

1. где проект остановился;
2. кто должен действовать;
3. что просрочено или блокирует работу;
4. куда перейти для следующего решения.

Клиент открывает платформу преимущественно из письма или с телефона. Он должен увидеть одно
объяснённое действие, выполнить его и получить явное подтверждение. Клиент не обязан понимать
структуру workspace, внутренние статусы или терминологию разработки.

Интерфейс должен ощущаться как спокойный стол передачи проекта: содержательный, собранный и
профессиональный, без визуального шума и шаблонного dashboard.

## 2. Информационная архитектура

### Владелец

- `Обзор` — состояние работы и следующие действия;
- `Проекты` — все проекты и создание нового;
- `Клиенты` — компании заказчиков;
- `Доступ` — участники workspace, приглашения и собственные сессии.

### Клиент

- `Главная` — следующее действие и состояние доступных проектов;
- `Проекты` — только проекты с явным membership.

### Внутри проекта

- `Обзор`;
- `План`;
- `Анкеты`;
- `Материалы`.

Milestones 07–10 добавляют новые пункты только при наличии реализованного и доступного содержимого.
Пустые будущие разделы не показываются.

## 3. Signature: маршрут проекта

Повторяемый элемент продукта отвечает на четыре вопроса:

1. какой этап или состояние актуально;
2. кто сейчас действует;
3. какое следующее действие;
4. что произойдёт после него.

Маршрут используется на workspace overview, project overview, в клиентском next-action block, в
списке проектов и позднее в review/approval/handover. Он не является декоративным progress bar:
каждый сегмент должен объяснять ответственность и переход.

## 4. Иерархия действий

- На экране один визуально доминирующий CTA.
- Вторичные действия оформляются нейтральными кнопками или ссылками.
- Опасные действия вынесены из основного пути и требуют явного подтверждения.
- После нажатия элемент немедленно показывает pending state и блокирует повторную отправку.
- Success объясняет результат и следующий шаг; error сохраняет введённые данные, где это возможно.

## 5. Состояния

Каждая пользовательская поверхность обязана иметь применимые состояния:

- loading/skeleton;
- empty с конкретным безопасным следующим действием;
- success;
- validation error рядом с полем;
- recoverable server/network error;
- permission denied без подтверждения существования чужого объекта;
- read-only/archive;
- pending external processing.

Фразы описывают ситуацию и действие пользователя. Не использовать stack trace, внутренние ID,
англоязычные enum или восклицательные success messages.

## 6. Визуальная система

- Тёплый светлый canvas и поверхности одного цветового мира.
- Графитовый текст четырёх уровней контраста.
- Глубокий зелёный — основное действие и подтверждённое движение.
- Янтарный — ожидание и внимание, не декор.
- Приглушённый красный — блокирующая ошибка или destructive action.
- Depth строится тихими borders и небольшим surface shift; тяжёлые тени и gradients не используются.
- Базовый spacing unit — 4 px.
- Небольшой radius у controls, средний у самостоятельных panels.
- Serif допустим только для крупных editorial headings; интерфейсные подписи и controls — sans-serif.
- Числа прогресса и сроки используют tabular numerals.

## 7. Производительность

Цели проверяются на production build отдельно от development cold compilation:

- визуальная реакция на действие — до 100 мс;
- повторная внутренняя навигация обычно начинает переход без заметной паузы;
- обычная server-rendered страница локально стремится к ответу до 1 секунды;
- мутация без внешнего processing стремится к 500–800 мс;
- scan, email и другие внешние операции остаются асинхронными и показывают понятное состояние.

Оптимизация выполняется по измерениям: параллельные независимые queries, отсутствие N+1, Next Link
prefetch, route loading UI, минимизация full-page POST redirects и безопасное кэширование только
нечувствительных данных.

## 8. Accessibility и mobile

- Skip link присутствует на каждой странице.
- Все действия доступны с клавиатуры и имеют заметный focus.
- Навигация имеет `aria-label` и `aria-current`.
- Touch targets не меньше 44×44 px.
- Значение не передаётся только цветом.
- Mobile layout сохраняет главное действие первым и не создаёт горизонтальную прокрутку.
- Motion отключается или упрощается при `prefers-reduced-motion`.
- Критические пути проходят axe smoke и keyboard review.

## 9. Критерии Milestone 06.5

- Неавторизованный экран объясняет продукт и ведёт к одному действию.
- Действующая сессия сохраняется при навигации и повторном открытии приложения.
- Владелец и клиент получают разные overview и навигацию.
- Ключевые функции Milestones 01–06 доступны не более чем за два осмысленных перехода.
- Project overview показывает маршрут, прогресс, ответственность и следующее действие.
- Все основные forms имеют pending/success/error и защиту от повторной отправки.
- Production performance измерено до и после; development cold compilation отмечена отдельно.
- Desktop/mobile E2E и axe проверяют реальное поведение.
- Server policies, tenant isolation, audit и quarantine не ослаблены.

## 10. Не входит

- бизнес-функции Milestone 07;
- dark mode;
- real-time;
- маркетинговый лендинг;
- сложные анимации;
- публичный SaaS onboarding;
- ослабление security ради уменьшения количества шагов.

## 11. Результат внедрения

- Public entry теперь объясняет назначение продукта и предлагает одно главное действие.
- Вход разделён на два явно переключаемых способа; пароль выбран по умолчанию, magic link остаётся
  безопасным вариантом для приглашённых пользователей.
- Общий tenant-aware shell сохраняет контекст workspace и показывает разную навигацию владельцу и
  клиенту. На mobile основные разделы доступны в нижней навигации.
- Overview владельца начинается с одного следующего шага и показывает проекты, ожидающие клиента.
  Клиентский overview показывает только назначенные проекты и ближайшее доступное действие.
- `ProjectRoute` показывает состояние, ответственность, следующий результат и реальный прогресс.
- Редко используемые настройки проекта, управление участниками, приглашение клиента и архив
  раскрываются по запросу и не конкурируют с основным маршрутом.
- Формы входа, приглашений, создания клиента/проекта и административных действий блокируют повторную
  отправку и показывают pending state.
- Все новые entry/auth страницы имеют skip target; прогресс использует нативный `progress`; цветовые
  токены прошли axe contrast check на desktop и mobile.

### Измерения

До оптимизации development cold compilation составляла 18,6 с для `/` и 10,5 с для первого
workspace route. Это время преимущественно относится к первичной компиляции Next.js dev server и не
используется как production SLA.

На production build Playwright-сценарии public entry/login выполняют загрузку и проверку поведения за
0,6–2,4 с вместе с browser automation; полные stateful сценарии identity, project, questionnaire и
material проходят за 11–18 с каждый, включая реальные PostgreSQL, Redis, Mailpit, MinIO и ClamAV.
Повторная навигация использует `Link`, route loading UI и кэшированное разрешение tenant в рамках
render. Отдельный RUM/APM baseline остаётся задачей production-наблюдаемости после выбора hosting.
